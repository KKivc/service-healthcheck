## 1. 背景与机会

### 1.1 市场现状

当前服务监控市场呈现明显的两极分化：

| 方案 | 优势 | 劣势 |
|------|------|------|
| **UptimeRobot** | 免费额度 50 个监控器，配置简单 | 只能检查 HTTP 状态码，无自定义诊断，无延迟趋势分析 |
| **Pingdom** | 功能全面，全球多节点 | 价格高（$15/月起），面向企业 |
| **Prometheus + Grafana** | 强大可定制，指标丰富 | 部署复杂，学习成本高，对小项目过于沉重 |
| **自写脚本** | 完全自由 | 无可视化面板，无告警体系，难以维护 |

### 1.2 机会点

> **"我只需要 5 分钟部署一个监控面板，告诉我网站挂了没有，仅此而已。"**

这是个人开发者和小团队的普遍诉求。现有方案要么太贵、要么太重、要么什么都没有。市场存在一个明确的空缺：**面向个人开发者的、开箱即用的轻量级健康监控面板**。

### 1.3 产品定位

**一句话定位：** 5 分钟部署、零配置、带 AI 诊断的轻量级服务监控面板。

## 3. 用户故事

### Epic 1：服务监控

| ID | Story | Acceptance Criteria |
|----|-------|-------------------|
| US1.1 | 作为用户，我想添加一个服务的 URL 和名称，让它被自动监控 | 输入名称+URL 即可添加；支持 HTTP/HTTPS；支持 ping 协议 |
| US1.2 | 作为用户，我想在仪表盘上一眼看到所有服务的实时状态 | 绿色/红色一目了然；表格显示状态码、延迟、上次检查时间 |
| US1.3 | 作为用户，我想看到服务的历史延迟趋势 | 曲线图展示每个服务的延迟变化；支持查看任意时间段 |

### Epic 2：故障告警

| ID | Story | Acceptance Criteria |
|----|-------|-------------------|
| US2.1 | 作为用户，服务挂了我想立刻收到通知 | 支持飞书/钉钉 webhook；仅在状态变更时告警（防重复） |
| US2.2 | 作为用户，我想知道服务为什么挂了 | AI 自动诊断错误原因并给出修复建议；支持多种 LLM 后端 |

### Epic 3：用户体验

| ID | Story | Acceptance Criteria |
|----|-------|-------------------|
| US3.1 | 作为用户，我想在手机上也能正常查看监控面板 | 布局自适应移动端；表格可横向滚动 |
| US3.2 | 作为用户，我想要深色模式，夜间查看不刺眼 | 一键切换深色；记住偏好设置 |
| US3.3 | 作为用户，我想对服务进行增加和删除操作 | 前端直接操作，无需编辑配置文件 |

## 4. 功能清单

### P0 — 核心功能（MVP）

- [x] 服务注册：通过 UI 添加/删除被监控服务
- [x] 健康检查：定时对每个服务发起 HTTP 请求
- [x] 状态看板：表格展示所有服务的当前状态
- [x] 状态卡片：总数/正常/异常一目了然
- [x] 延迟记录：每次检查记录响应时间
- [x] 延迟趋势图：Chart.js 折线图展示历史延迟

### P1 — 重要功能

- [x] 异常告警：状态变更时通过 webhook 推送飞书/钉钉
- [x] AI 诊断：服务异常时自动调用 LLM 分析根因
- [x] 健康评分：综合延时 + 历史故障计算健康分（0-100）
- [x] 深色模式：一键切换 + localStorage 持久化
- [x] 数据清理：自动保留最近 100 条记录

### P2 — 锦上添花

- [ ] 多节点监控：从不同地域同时检查
- [ ] 通知渠道扩展：邮件、短信、企业微信
- [ ] 用户系统：多用户登录、权限管理
- [ ] 公开展示模式：生成公开状态页（如 status.example.com）
- [ ] 自定义检查间隔：按服务单独配置频率

### 功能取舍说明

> **为什么不做用户系统？**
> MVP 阶段面向个人/小团队使用场景，用户系统只会增加部署复杂度。如果未来发现多人协作是强需求，再引入轻量级认证。

> **为什么选择 Chart.js 而不是 ECharts？**
> Chart.js 体积更小（~70KB vs ~800KB），CDN 加载快，对于延迟折线图这个单一场景功能完全够用。

## 5. 成功指标

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| **部署时间** | < 5 分钟 | 从 git clone 到看到面板 |
| **面板加载速度** | < 2 秒 | Lighthouse / 实际测速 |
| **异常检出时间** | < 检查间隔 + 30 秒 | 从服务挂掉到看板更新 |
| **AI 诊断准确率** | > 80% 建议有参考价值 | 用户反馈 + 人工评估 |

## 6. 技术架构

```
┌─────────────────┐     ┌──────────────┐     ┌───────────────┐
│  健康检查引擎     │────▶│   SQLite     │◀────│  Flask Web    │
│  (healthcheck.py)│     │  数据库      │     │  (web.py)     │
└────────┬────────┘     └──────────────┘     └───────┬───────┘
         │                                            │
         ▼                                            ▼
┌─────────────────┐                       ┌──────────────────┐
│  Webhook 告警    │                       │  HTML + Chart.js │
│  (飞书/钉钉)      │                       │  (浏览器渲染)     │
└─────────────────┘                       └──────────────────┘
         │                                            ▲
         ▼                                            │
┌─────────────────┐                                   │
│  LLM 诊断引擎    │───────────────────────────────────┘
│  (OpenAI 兼容)   │  AI 诊断文本写入数据库，前端展示
└─────────────────┘
```

**技术栈选择理由：**

| 选择 | 替代方案 | 理由 |
|------|----------|------|
| **Flask** | FastAPI / Django | 项目只有一个页面，Flask 零开销。FastAPI 的异步优势在此场景无意义 |
| **SQLite** | MySQL / PostgreSQL | 个人项目无需并发写入，SQLite 零运维、文件即数据库 |
| **Chart.js** | ECharts / D3 | 一个折线图场景，Chart.js CDN 仅 70KB |
| **requests** | httpx / aiohttp | 同步检查即可，不需要异步 |
